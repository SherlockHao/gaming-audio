# AI 游戏音频生产与集成系统 PRD v0.1

## 1. 产品信息

**产品名称**  
AI 游戏音频生产与集成系统

**文档版本**  
v0.1

**文档状态**  
内部讨论稿

**文档目标**  
定义一套面向游戏研发中前期的 AI 音频生产系统。该系统由专业音频人员在项目初期完成规则、模板与流程建制，之后在较长开发周期内，支持策划和相关制作人员通过低门槛输入触发标准化音频生产、Wwise 集成、UE 挂接和自动化 QA 诊断，以最小化中前期音频人力投入，并为后期正式音频量产与精修保留衔接空间。

---

## 2. 产品背景

在长周期游戏项目中，音频团队往往需要较早介入制定风格、标准与系统架构，但在项目中前期，大量需求属于标准化、重复性、高迭代频次的音频工作，例如动作配音效、UI 声音、机关声音、基础环境循环、Wwise 资源导入、事件创建、UE 挂接、问题排查等。

### 2.1 当前痛点

- 策划或相关制作人员不会专业音频描述，提需求成本高。
- 音频人员在中前期被大量标准化生产和接入工作占用。
- Wwise 工程结构、命名、导入、Bank 管理和 UE 挂接容易不一致。
- 游戏内出现“没声”“播错声”“音量异常”“被错误 kill”等问题时，定位链路长、成本高。
- 项目后期需要大规模替换和精修时，前期资产若缺少标准和追踪能力，会造成返工。

### 2.2 产品机会

- 由资深音频人员在前期定义风格、规则、模板和 AI 流程。
- 将中前期大量标准化音频工作转化为 AI 驱动流水线。
- 将音频生成、Wwise 工程、UE 接入和 QA 诊断打通为一个闭环。
- 让项目在前三年左右的开发迭代期以极低人力维持音频可用状态，后期再由正式音频团队接管高质量迭代和终混。

---

## 3. 产品定位

本产品不是“让非音频人员替代音频团队”，而是“由专业音频前置建制后，利用 AI 接管中前期标准化音频生产和集成工作”。

产品分为三阶段使用模式：

1. **音频前置建制期**  
   由专业音频人员建立风格规范、类别标准、Wwise 模板、UE mapping 规则、AI 工作流和 QA 规则。
2. **AI 量产迭代期**  
   项目中前期由策划或相关制作人员提供低门槛输入，系统自动完成大部分标准化音频工作。
3. **人工精修收尾期**  
   项目后期由正式音频团队接管高质量资源替换、复杂实时系统设计与调试、终混和问题收敛。

---

## 4. 产品目标

### 4.1 核心目标

- 降低中前期音频生产与接入的人力成本。
- 让非音频专业角色以低门槛方式触发标准化音频流程。
- 建立从需求输入到游戏内验证的端到端自动化链路。
- 实现问题可追踪、可回滚、可诊断。

### 4.2 阶段目标

- 第一阶段只覆盖高可行部分，先打通完整流程。
- 第二阶段扩展中可行能力，如更复杂的交互逻辑和语义匹配。
- 不覆盖低可完全自动化的高艺术性与高复杂度音频工作。

---

## 5. 非目标

以下内容不在 MVP 范围内：

- 高品质商业级原创音乐成品制作
- 复杂互动音乐系统自动设计
- 高质量角色配音导演与表演控制
- 项目终混自动完成
- 复杂实时音频系统的全自动设计与深度调优
- 纯 AI 黑盒全自动无人工审批的正式量产流程

---

## 6. 目标用户

### 6.1 主要用户

- **音频总监/高级音频设计师**  
  负责前期风格、规则、模板、审批与后期接管。
- **音频技术设计师/TA**  
  负责系统配置、Wwise 模板、UE mapping、自动化流程维护。
- **策划/关卡/战斗策划**  
  提交动作、技能、UI、机关等音频需求。
- **QA**  
  利用自动化和人工复盘能力验证问题、反馈异常。

### 6.2 次要用户

- 程序  
  查看接入日志、定位触发和通路问题。
- 制作人/项目管理  
  查看产出效率、问题率、自动化覆盖率。

## 7. 核心使用场景

### 7.1 动作配音效

策划上传一个角色攻击动作视频，标注“Boss 技能，一次性播放”，系统自动完成视频理解、SFX 生成、规则检测、Wwise 导入、Event 创建、Bank 生成、UE 挂接和基础 QA。

### 7.2 UI 音频批量生产

策划提交一批 UI 交互需求及功能名称，系统自动按 UI 类别规范生成、处理并导入。

### 7.3 机关/环境循环音

策划提供机关动画或场景视频，并标注循环播放，系统自动生成 loop 类资产并接入 Wwise 与 UE。

### 7.4 问题诊断

QA 反馈“第 132 秒 Boss 二段砸地没有声音”，系统自动查询对应 profile 记录，定位是 Event 未触发、Voice 未创建、RTPC 拉为 0、Bus 静音还是资源未加载。

---

## 8. MVP 范围

### 8.1 纳入范围

- 资产类型：SFX、UI、简单环境循环
- 输入形式：视频/动画资源 + 场景语义 + 播放方式
- 生成形式：多候选音频生成
- 后处理：分类响度与频段统一、基础清洗
- Wwise：模板派生、对象创建、素材导入、Event 创建、Bank 生成
- UE：AnimNotify 挂接、基础事件绑定
- QA：自动化流程跑测、profile 抓取、基础问题诊断
- 审批：音频审批、挂接审批
- 追踪：任务级记录、改动清单、回滚点

### 8.2 暂不纳入

- VO
- 复杂音乐系统
- 复杂 Switch/State/RTPC 自动设计
- 复杂蓝图逻辑自动接入
- 最终混音智能决策

## 9. 总体流程

1. 专业音频人员完成项目风格、规则、模板和流程建制。
2. 策划提交低门槛需求。
3. 系统进行视频理解与需求结构化。
4. 系统根据规则库生成 Audio Intent Spec。
5. 系统生成音频候选素材。
6. 系统进行基础检测与后处理。
7. 系统按模板规则自动派生 Wwise 结构并导入资产。
8. 系统生成 Event、Bank 和相关清单。
9. 系统在 UE 中按 mapping 规则自动挂接。
10. 系统触发自动化 QA 流程并抓取 profile。
11. 系统输出结果、日志、问题报告和审批项。
12. 审批通过后进入项目资源库；若失败则回滚并重新生成或修正。

## 10. 模块设计

### 10.1 低门槛需求输入层

设计原则：

- 不要求策划使用专业音频词汇。
- 不要求策划决定分层、容器结构、参数方案。
- 只要求策划描述游戏内容和基础语义。

输入字段：

- 任务名称
- 输入素材
- 资源类型：动作 / 技能 / 特效 / UI / 机关 / 环境
- 场景语义：小怪 / Boss / 机关 / 场景交互 / 系统 UI
- 播放方式：一次性 / 循环
- 可选标签：轻 / 重 / 稀有 / 魔法 / 金属 / 近战 / 远程
- 优先级
- 可选备注

输入素材形式：

- 视频
- 动画资源引用
- UI 操作录屏
- 美术特效预览
- 资源名和路径

系统输出：

- 结构化需求对象
- Audio Intent Spec
- 置信度评分
- 待确认项

**Audio Intent Spec 核心字段**

- task_id
- content_type
- semantic_role
- play_mode
- intensity
- material_hint
- timing_points
- variation_count
- design_pattern
- category_rule_id
- wwise_template_id
- ue_binding_target
- confidence

### 10.2 风格与规则库层

该层由专业音频人员在项目初期建立，是系统的核心基础能力。

规则库组成：

- **Style Bible**  
  定义项目听觉风格、参考作品、关键词、禁用倾向。
- **Category Rules**  
  按类别定义格式与基础声学标准。
- **Implementation Templates**  
  定义 Wwise 和 UE 接入模板。
- **Mapping Dictionary**  
  定义资源名、动作名、技能 ID、角色类别与音频语义之间的映射关系。

**Category Rules 示例**

- SFX：48kHz、24-bit、单声道优先、响度区间、瞬态完整性、尾部长度规范
- UI：短时长、清晰瞬态、频段避让、统一响度
- Ambience Loop：循环无点击、尾部衔接、立体声策略

**Implementation Templates 示例**

- Actor-Mixer Hierarchy 模板
- Master-Mixer Hierarchy 模板
- Event 命名模板
- Bus 指派规则
- SoundBank 切分规则
- Conversion Setting 规则

### 10.3 素材生成与后处理层

MVP 目标为基础可用，不追求复杂母带能力。

核心能力：

- 根据 Audio Intent Spec 生成多个候选音频
- 自动切除无效头尾
- 自动基础淡入淡出
- 按类别统一响度
- 按类别执行基础频段检测
- 输出导入版素材

输入：

- Audio Intent Spec
- Style Bible
- Category Rules
- 输入视频或动画参考

输出：

- Source 文件
- Processed 文件
- Mastered 文件
- QC Report

MVP 检测指标：

- 峰值
- 基础响度区间
- 静音头尾
- 时长范围
- 声道数
- 采样率
- 位深
- 基础频谱异常

### 10.4 Wwise 自动化层

设计原则：

- 不是从零自动设计完整 Wwise 工程。
- 是从“项目模板 + 游戏类型模板 + 类别规则”派生实际结构。

核心能力：

- 自动创建或补全 Actor-Mixer Hierarchy
- 自动创建或补全 Master-Mixer Hierarchy
- 自动导入处理后音频
- 自动创建 Event
- 自动创建或更新 Bank
- 自动设置命名、Bus、Conversion Setting、基础参数
- 输出构建和导入日志

模板类型：

- 动作游戏模板
- RPG 模板
- 开放世界模板
- 卡通休闲模板

MVP 自动化对象：

- Sound SFX
- Random Container
- 基础 Actor-Mixer
- 基础 Bus
- Event
- SoundBank

MVP 暂不自动化的高级对象：

- 复杂 Music Segment/Playlist
- 复杂 State/Switch 体系
- 高复杂 RTPC 设计
- 高级混音 Snapshot 体系

Wwise 输出物：

- Wwise object manifest
- import manifest
- bank manifest
- build log

### 10.5 UE 自动挂接层

设计原则：

- 优先规则驱动，辅以语义匹配。
- 置信度不足时进入人工确认队列。

挂接策略：

- **强规则匹配**  
  根据动画名、技能 ID、角色类型、资源路径直接匹配 Event。
- **弱语义匹配**  
  根据输入视频和动作语义推断应挂接的 Event。
- **人工确认**  
  对低置信度项进行审核。

挂接目标：

- AnimNotify
- AkComponent
- Blueprint/C++ 调用包装

MVP 范围：

- 动画触发类声音通过 AnimNotify 自动挂接
- 简单持续类声音绑定基础组件
- 生成 UE 挂接清单和差异日志

UE 输出物：

- binding manifest
- notify mapping report
- unresolved binding queue

### 10.6 审批、追踪与回滚层

设计目标：

- 所有自动生成和自动修改都必须可追踪。
- 所有关键阶段都必须具备回滚点。

记录对象：

- 需求 ID
- 提交人
- 输入素材版本
- 规则版本
- AI 模型与参数
- 生成音频版本
- Wwise 改动清单
- UE 改动清单
- QA 结果
- 审批记录
- 回滚点

审批节点：

- 音频产出审批
- 接入挂接审批

回滚能力：

- 回滚到上一个素材版本
- 回滚到上一个 Wwise 对象状态
- 回滚到上一个 UE 绑定状态

### 10.7 自动化 QA 与诊断层

这是系统闭环的关键模块。

**模块目标**

- 自动检测音频是否按预期触发和输出
- 自动定位问题发生在链路的哪个环节
- 生成可读的诊断报告供音频、程序、QA 协作

**接入方式**

- 接入现有自动脚本跑游戏流程
- 联动 Wwise profiling
- 自动抓取 profile 数据和运行时日志
- 支持人工指定时间点和问题对象进行复盘

**规则型 QA 检测内容**

- Event 是否触发
- 声音是否真正发声
- Voice 是否被 kill 或 virtualize
- 音量是否异常为 0
- RTPC 是否超出合理范围
- Bus 路由是否错误
- Bank 是否加载
- Media 是否缺失
- 是否命中错误对象或错误 Event

**诊断型 QA 流程**

1. 接收问题描述，例如“132 秒 Boss 二段砸地没声音”
2. 根据时间点定位 profile capture
3. 根据角色名、技能名、资源名映射到目标 Wwise 对象
4. 检查 Event 是否触发
5. 检查 Voice 是否创建
6. 检查 Voice 是否被限制、虚拟化或 kill
7. 检查音量贡献链路
8. 检查 RTPC、State、Switch、Bus、Bank、Media 状态
9. 输出疑似根因和建议修复动作

**QA 输出格式**

- 问题 ID
- 发生时间
- 关卡/角色/技能
- 相关 Event
- 相关对象路径
- 问题类型
- 疑似根因
- 建议修复动作
- 证据链接

**问题类型示例**

- Event 未触发
- Event 触发但无 Voice
- Voice 被错误 Kill
- RTPC 导致音量降为 0
- Bus Mute 或音量异常
- State/Switch 条件不成立
- Bank 未加载
- Media 缺失
- 映射对象错误

## 12. 核心流程示例

### 12.1 流程 A：动作音效自动生产

1. 策划上传动作视频并选择“Boss 技能”“一次性播放”。
2. 系统识别动作节奏、打击点、尾韵和能量感。
3. 系统生成 Audio Intent Spec。
4. 系统生成 3 个 SFX 候选。
5. 系统执行响度和频段检查并输出导入版。
6. 系统在 Wwise 模板下创建对象、导入音频、创建 Event、生成 Bank。
7. 系统在 UE 中根据动画资源名挂接 AnimNotify。
8. 自动化脚本跑一轮技能播放测试并抓 profile。
9. 若无异常，进入待审批状态。
10. 审批通过后入库。

### 12.2 流程 B：QA 问题诊断

1. QA 提交问题：“132 秒 Boss 二段砸地没声音”。
2. 系统提取该时间点 profile 数据。
3. 系统根据 Boss 和技能标识定位目标 Event。
4. 系统判断 Event 已触发但 Voice 未最终输出。
5. 系统检查发现 RTPC 在该时刻将音量拉至 0。
6. 系统输出诊断报告与建议修复动作。
7. 系统可自动修正规则或进入人工处理队列。

## 15. 依赖条件

- 项目已有基本音频风格方向
- 音频团队能投入前期建制工作
- Wwise 工程与 Unreal 集成链路可用
- 项目具备自动化跑测基础
- 能获取运行时 profile 和日志
- 项目资源命名和目录结构具备一定规范性

---

## 16. 风险与边界

- 若前期规则库建设不完整，后续自动化质量会显著下降。
- 若项目资源命名混乱，UE 自动挂接准确率会受到影响。
- 若输入视频质量差或动作表现不明确，音频意图推断会不稳定。
- 资产级响度标准化不能替代最终混音。
- AI 生成素材适合作为中前期可用版本，不应默认等同于最终量产版本。
- QA 诊断依赖 profiling 数据质量和映射关系完整度。
- Wwise 模板若设计不合理，后续将放大工程结构问题。
